Duration, Effort, Size - Difference in scales

There exist several scales which can be used to estimate development work. Everyone of them can be used for a specific purpose and is not interchangeable. Although they can be transformed to each other with a bit of magic.

Estimating the duration of something

How long does it take?

This is often the type of estimation Management (with a big M intentionally) wants when they asked for estimations. The fun part is that the answer most of the time is: It depends. The reason for that will get clearer when we have a look at the parameters influencing duration:

  • Size of the task
  • Amount of working entities working on the task (minus communication overhead)
  • Skill of working entities working on the task (how much effort can they pull of per time unit)
  • Unproductive/Wait time
  • Time spent on other tasks

So in order to create a good enough estimate for the duration of a task, the parameters need to be frozen at the time of estimation. This is a quite difficult constraint for long term tasks.

Example for duration

Let's model the question "How long does it take to transport one tiny package from Vienna to Berlin?":

  • Size of the task: ~ 700 km
  • Amount of working entities working on the task (minus communication overhead): one car
  • Skill of working entities working on the task (how much effort can they pull of per time unit): 100 km/h
  • Unproductive/Wait time: 2 h (heavy traffic jam)
  • Time spent on other tasks: 1 h (one detour to Prague)

So the duration will be 10 hours. This estimation is highly dependable on multiple parameters. If only one of them changes the whole estimation is off (a lot).

Estimating the effort of something

How much does it cost? There is one parameter describing the effort:
  • Skill of working entities working on the task

This means if there is a difference between the estimator and the implementer this will have a (big) impact on the estimation. Here are some examples:

Junior ProgrammerSenior Programmer

Estimating the size of something

How big is it? Absolute size is very hard to estimate if the numbers get bigger. It is easy to estimate if something is one or two meters tall. I will have a very hard time to estimate the height of the skyscraper over there. Comparing two skyscrapers standing next to each other is a lot easier. In a software project an absolute size would be:
  • Changed classes
  • Lines of code added
  • Function points created
  • Acceptance criteria implemented
